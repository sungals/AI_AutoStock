## 3. Gunicorn 설정

### 3.1 Gunicorn 실행

```bash
cd /home/myuser/my_quant/app

venv/bin/gunicorn web_app:app \
  -b 127.0.0.1:5000 \
  --workers 2 \
  --timeout 120 \
  --daemon \
  --log-file /home/myuser/my_quant/logs/gunicorn.log \
  --access-logfile /home/myuser/my_quant/logs/access.log \
  --pid /tmp/gunicorn.pid
```

**workers=2 권장 이유**: SQLite WAL 모드는 다중 writer를 지원하지만,
과도한 동시 쓰기는 `busy_timeout`을 자주 유발합니다.
workers=2는 성능과 안정성의 균형점입니다.

**timeout=120**: 10,000회 시뮬레이션 등 장시간 요청을 위해 충분한 timeout을 설정합니다.

### 3.2 Systemd 서비스 등록 (권장)

```ini
# /etc/systemd/system/my-quant.service

[Unit]
Description=My Quant Flask App
After=network.target

[Service]
User=myuser
WorkingDirectory=/home/myuser/my_quant/app
Environment="PATH=/home/myuser/my_quant/app/venv/bin"
ExecStart=/home/myuser/my_quant/app/venv/bin/gunicorn web_app:app \
    -b 127.0.0.1:5000 \
    --workers 2 \
    --timeout 120 \
    --log-file /home/myuser/my_quant/logs/gunicorn.log \
    --access-logfile /home/myuser/my_quant/logs/access.log
ExecReload=/bin/kill -s HUP $MAINPID
ExecStop=/bin/kill -s TERM $MAINPID
Restart=always
RestartSec=5

[Install]
WantedBy=multi-user.target
```

```bash
sudo systemctl daemon-reload
sudo systemctl enable my-quant
sudo systemctl start my-quant
sudo systemctl status my-quant
```

### 3.3 Gunicorn 재시작

```bash
# PID 파일로 재시작
kill -HUP $(cat /tmp/gunicorn.pid)

# 또는 직접 재시작
pkill -f gunicorn; sleep 2
cd /home/myuser/my_quant/app
venv/bin/gunicorn web_app:app -b 127.0.0.1:5000 --workers 2 --timeout 120 --daemon \
    --log-file /home/myuser/my_quant/logs/gunicorn.log
```

## 4. Nginx 설정

### 4.1 기본 리버스 프록시 (루트 경로)

```nginx
# /etc/nginx/sites-available/my-quant

server {
    listen 80;
    server_name your-domain.com;   # 또는 IP 주소

    # Gunicorn으로 프록시
    location / {
        proxy_pass http://127.0.0.1:5000;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;

        # SSE를 위한 버퍼링 비활성화
        proxy_buffering off;
        proxy_cache off;
        proxy_read_timeout 300s;    # 시뮬레이션 등 장시간 요청
    }

    # 정적 파일 직접 서빙 (성능 개선)
    location /static/ {
        alias /home/myuser/my_quant/app/static/;
        expires 1d;
    }
}
```

### 4.2 서브 경로 배포 (예: /ttakquant)

서브 경로로 배포하는 경우, Nginx에서 prefix를 제거하고 Flask에 전달합니다.

```nginx
# /etc/nginx/sites-available/my-quant

server {
    listen 80;
    server_name your-domain.com;

    location /ttakquant {
        # prefix 제거 후 Gunicorn으로 전달
        rewrite ^/ttakquant(/.*)$ $1 break;
        rewrite ^/ttakquant$ / break;

        proxy_pass http://127.0.0.1:5000;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_buffering off;
        proxy_read_timeout 300s;
    }

    location /ttakquant/static/ {
        alias /home/myuser/my_quant/app/static/;
        expires 1d;
    }
}
```

Flask `.env`에 설정:

```
BASE_PATH=/ttakquant
```

이 값은 Jinja2 템플릿에서 `{{ config.BASE_PATH }}`로 접근하여 URL을 생성할 때 사용합니다.

### 4.3 HTTPS 설정 (Let's Encrypt)

```bash
sudo apt install -y certbot python3-certbot-nginx
sudo certbot --nginx -d your-domain.com
```

Certbot이 자동으로 Nginx 설정을 수정하여 HTTPS를 설정합니다.

### 4.4 Nginx 활성화

```bash
sudo ln -s /etc/nginx/sites-available/my-quant /etc/nginx/sites-enabled/
sudo nginx -t         # 설정 문법 확인
sudo systemctl reload nginx
```

## 5. Cron 설정 (일일 파이프라인)

### 5.1 로그 디렉토리 생성

```bash
mkdir -p /home/myuser/my_quant/logs
```

### 5.2 crontab 등록

```bash
crontab -e
```

아래 내용 추가 (절대 경로 사용, `/home/myuser/my_quant` 실제 경로로 변경):

```cron
# ─── TTAK Quant 자동 실행 ───────────────────────────────────────

# 매크로 지표 hourly 수집 (평일)
0 * * * 1-5 cd /home/myuser/my_quant/app && venv/bin/python update_macro.py \
    >> /home/myuser/my_quant/logs/macro_cron.log 2>&1

# EOD 파이프라인 — 평일 16:00 KST (07:00 UTC)
# 장마감 후 자동으로 OHLCV, 재무, 뉴스, 스크리닝, 시뮬레이션 실행
0 7 * * 1-5 cd /home/myuser/my_quant/app && venv/bin/python run_daily_pipeline.py \
    >> /home/myuser/my_quant/logs/cron.log 2>&1

# Full 파이프라인 — 일요일 16:00 KST (종목 마스터 갱신 포함)
0 7 * * 0 cd /home/myuser/my_quant/app && venv/bin/python run_daily_pipeline.py \
    >> /home/myuser/my_quant/logs/cron.log 2>&1
```

**KST ↔ UTC 변환**: 한국 표준시(KST, UTC+9) 16:00 = UTC 07:00

### 5.3 cron 동작 확인

```bash
# 다음 날 로그 확인
tail -f /home/myuser/my_quant/logs/cron.log

# 수동 파이프라인 실행 테스트
cd /home/myuser/my_quant/app
venv/bin/python run_daily_pipeline.py --dry-run  # 실제 API 호출 없이 테스트
```

## 6. 배포 스크립트 (deploy.sh)

코드 업데이트 및 재시작을 자동화하는 배포 스크립트 예시입니다.

```bash
#!/bin/bash
# deploy.sh — 서버 코드 업데이트 + Gunicorn 재시작

set -e

BRANCH="${1:-main}"
APP_DIR="/home/myuser/my_quant"

echo "[deploy] Branch: $BRANCH"

# 서버에 접속하여 배포 실행
ssh myuser@your-server-ip << ENDSSH
    cd $APP_DIR

    # 최신 코드 가져오기
    git fetch origin
    git checkout $BRANCH
    git pull origin $BRANCH

    # 의존성 업데이트 (변경 사항 있을 때만 실질적으로 시간 소요)
    cd app
    venv/bin/pip install -r requirements.txt --quiet

    # Gunicorn 재시작
    pkill -f gunicorn || true
    sleep 2
    venv/bin/gunicorn web_app:app \
        -b 127.0.0.1:5000 \
        --workers 2 \
        --timeout 120 \
        --daemon \
        --log-file $APP_DIR/logs/gunicorn.log \
        --access-logfile $APP_DIR/logs/access.log

    echo "[deploy] 완료"
ENDSSH
```

```bash
chmod +x deploy.sh
./deploy.sh main     # main 브랜치 배포
./deploy.sh dev      # dev 브랜치 배포
```

## 7. 보안 체크리스트

배포 전 반드시 확인합니다.

| 항목 | 확인 방법 |
|------|-----------|
| SECRET_KEY 설정 | `.env` 확인 (`python3 -c "import secrets; print(secrets.token_hex(32))"` 생성 사용) |
| .env 파일 권한 | `chmod 600 .env` (소유자만 읽기/쓰기) |
| .env gitignore | `.gitignore`에 `.env`, `*.db` 포함 확인 |
| DB 파일 권한 | `chmod 600 quant_data.db` |
| FLASK_ENV | `production` (개발 디버그 모드 비활성화) |
| SESSION_COOKIE_SECURE | HTTPS 환경에서 `True` 자동 설정 확인 |
| KIS 모드 | `KOREAINVESTMENT_MODE=mock` (live 전환은 명시적으로) |
| 방화벽 | 5000번 포트 외부 접근 차단 (Nginx를 통해서만 접근) |

```bash
# 방화벽 설정 (ufw)
sudo ufw allow 80/tcp
sudo ufw allow 443/tcp
sudo ufw allow 22/tcp    # SSH
sudo ufw deny 5000/tcp   # Gunicorn 직접 접근 차단
sudo ufw enable
```

## 9. DB 백업 전략

### 9.1 SQLite WAL 안전 백업

SQLite WAL 모드에서는 `cp` 직접 복사 대신 `.dump`를 사용합니다.

```bash
# 안전한 백업 방법 1: sqlite3 .dump
sqlite3 /home/myuser/my_quant/app/quant_data.db ".dump" | \
    gzip > /home/myuser/backups/quant_data_$(date +%Y%m%d).sql.gz

# 안전한 백업 방법 2: sqlite3 .backup (온라인 백업)
sqlite3 /home/myuser/my_quant/app/quant_data.db \
    ".backup /home/myuser/backups/quant_data_$(date +%Y%m%d).db"
```

### 9.2 자동 백업 cron

```cron
# 매일 새벽 3시 DB 백업 (KST 03:00 = UTC 18:00 전날)
0 18 * * * sqlite3 /home/myuser/my_quant/app/quant_data.db \
    ".backup /home/myuser/backups/quant_data_$(date +\%Y\%m\%d).db" && \
    find /home/myuser/backups -name "*.db" -mtime +7 -delete
    # 7일 이상 된 백업 자동 삭제
```

### 9.3 복원

```bash
# sql.gz 복원
gunzip -c /home/myuser/backups/quant_data_20260305.sql.gz | \
    sqlite3 /home/myuser/my_quant/app/quant_data_restored.db

# .db 파일 복원
cp /home/myuser/backups/quant_data_20260305.db \
   /home/myuser/my_quant/app/quant_data.db
```

## 10. DB 경로 주의사항

`DB_PATH`와 `OUTPUT_DIR`은 `.env`에서 특별한 이유가 없으면 설정하지 않습니다.

```python
# config.py — 기본값이 항상 올바른 경로
DB_PATH = os.environ.get('DB_PATH', os.path.join(BASE_DIR, 'quant_data.db'))
```

`BASE_DIR`은 `config.py` 파일의 절대 경로를 기준으로 자동 계산됩니다.
잘못된 경로를 `.env`에 설정하면 새 빈 DB를 생성하여 로그인 불가 버그가 발생합니다.

```bash
# pykrx는 KRX 서버 상태에 따라 오류 발생
```

### 문제: DART API 한도 초과

```bash
# DB에서 오늘 사용량 확인
sqlite3 quant_data.db "SELECT * FROM api_usage WHERE usage_date=date('now');"

# api_usage 테이블의 provider='opendart', calls_made >= 10000이면 한도 초과
# → 다음 날 자동으로 리셋됨
```

### 문제: SQLite database is locked

```bash
# 이미 다른 프로세스가 쓰기 중인 경우
# busy_timeout=30000 (30초)으로 자동 재시도함
# 30초 후에도 지속되면:

# 현재 DB 잠금 상태 확인
sqlite3 quant_data.db ".tables"

# WAL 파일 정리 (안전, DB 닫힌 상태에서)
sqlite3 quant_data.db "PRAGMA wal_checkpoint(FULL);"
```
