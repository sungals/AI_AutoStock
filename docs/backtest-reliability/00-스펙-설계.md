# 백테스트 신뢰성 레이어 — 스펙(설계)

> **문서 목적**: TTAK Quant 백테스트/시뮬레이션 엔진의 신뢰성을 떨어뜨리는 4대 구조적 편향을
> 점진적 레이어로 보정하기 위한 설계 명세서입니다.
> 이 문서는 단독으로 읽고 구현에 착수할 수 있도록 작성되었습니다 (cold-start 가능).
>
> - 작성일: 2026-06-19
> - 통합 방식: **Approach A — 기존 엔진에 점진적 신뢰성 레이어 추가**
> - 데이터 제약: **현재 보유 데이터 범위 내에서 최대한** (상장폐지 종목 백필은 비목표)
> - 관련 문서: `01-구현-플랜.md`(task별 TDD 계획), `codex-handoff/`(에이전트 핸드오프 가이드)

---

## 1. 배경 — 왜 필요한가

TTAK Quant는 `simulation_runner`가 10,000회 walk-forward 시뮬레이션을 돌리고, 그 결과로
`algo_optimizer`가 전략 파라미터를 자동 조정한다(파이프라인 Step 6~8). 그런데 현재 백테스트는
다음 4가지 편향으로 **수익률을 구조적으로 과대평가**한다. 이 위에서 자동 최적화를 돌리면
"거짓 자신감"이 누적되어 실거래에서 손실로 이어진다.

| # | 편향 | 현재 코드의 문제 | 결과 |
|---|------|------------------|------|
| 1 | **생존편향 (Survivorship)** | `simulation_runner`가 *현재 상장된* `companies`로 과거를 시뮬레이션. 상장폐지·관리종목 누락 | 수익률 과대평가 |
| 2 | **미래참조 (Look-ahead)** | 재무제표를 `bsns_year`로만 조회 → 실제 공시 전 데이터로 과거 스크리닝 | PER/ROE 전략 성과 부풀림 |
| 3 | **체결 비현실성** | `commission_rate`만 존재. 매도세·슬리피지·상한가 매수불가·거래정지·유동성 무시 | 체결 불가능한 수익 계상 |
| 4 | **과최적화 (Overfitting)** | 다중검정 보정 없이 파라미터 자동 fitting. OOS 검증 부재 | 노이즈에 적합 |

---

## 2. 목표 / 비목표

### 목표
- **미래참조 편향 제거(2)**: 재무 데이터에 실제 공시일(`disclosed_at`)을 부여하고, 백테스트 시점 `T`에서 `disclosed_at <= T`인 데이터만 사용한다.
- **체결 현실성 확보(3)**: 거래비용(수수료+매도세+슬리피지)과 체결 가능성(상한가/거래정지/유동성/호가단위)을 순수함수 모델로 반영한다.
- **과최적화 방지(4)**: In-Sample(IS)에서 파라미터를 학습하고 Out-of-Sample(OOS) 성과만 보고하며, Deflated Sharpe Ratio로 다중검정을 보정한다.
- **생존편향 정량화(1)**: 완전 제거 대신, 편향의 크기를 측정·경고하고 보고 수익률에 보수적 haircut을 적용한다.

### 비목표 (YAGNI)
- 상장폐지 종목의 과거 시세 전면 백필 (데이터 확보 비용 과다 — 별도 프로젝트)
- 분/틱 단위 시장미시구조 시뮬레이션 (일봉 기준 유지)
- 신규 백테스트 엔진 v2 재작성 (기존 엔진 재사용)
- 실거래 주문 로직 변경 (이 스펙은 백테스트/시뮬레이션 한정)

## 3. 아키텍처

기존 Layered Architecture(03-시스템-아키텍처.md)를 유지한다. Business Layer에 **순수함수
모듈 3개 + 데이터 게이트 1개 + 리포트 1개**를 추가하고, 기존 `backtester.py` /
`simulation_runner.py` / `algo_optimizer.py`는 이 레이어를 경유하도록 최소 수정한다.

```
Business Layer
  backtester.py ───────────┐ (수정: 데이터·체결을 게이트 경유)
  simulation_runner.py ────┤
  algo_optimizer.py ───────┤ (수정: 검증 통과 파라미터만 반영)
                           │
                           ├─▶ point_in_time.py     [신규] PIT 재무 데이터 게이트
                           ├─▶ execution_model.py   [신규] 체결·비용 (순수함수)
                           ├─▶ validation_harness.py[신규] IS/OOS · Deflated Sharpe
                           └─▶ bias_report.py        [신규] 생존편향 정량화·경고
Data Access Layer
  db_financial.py ──────────  (수정: disclosed_at 컬럼 read/write)
  db_simulation.py ─────────  (수정: oos/deflated 컬럼 저장)
```

**설계 원칙**: 신규 3개 핵심 모듈(`point_in_time`, `execution_model`, `validation_harness`)은
`portfolio_operations.py`처럼 **사이드이펙트 없는 순수함수** 위주로 작성한다(테스트 용이).
DB 접근이 필요한 부분만 `conn`을 주입받는다(Connection Injection 패턴, 03-문서 §11.1).

## 4. 컴포넌트 명세

### 4.1 `point_in_time.py` — 미래참조 차단 게이트

**책임**: 특정 시점 `T`에 "그때 실제로 알 수 있었던" 재무 데이터만 반환한다.

**핵심 함수**
```python
from typing import Optional, List, Dict

def get_financials_asof(conn, corp_code: str, as_of_date: str) -> List[Dict]:
    """as_of_date(포함) 시점까지 공시된 재무제표만 반환.

    financial_statements.disclosed_at <= as_of_date 인 행만.
    disclosed_at이 NULL이면 estimate_disclosed_at()의 보수적 lag로 추정.
    """

def estimate_disclosed_at(bsns_year: str, reprt_code: str) -> str:
    """공시일 미상 시 보수적 추정.

    - 사업보고서(11011, 연간): 회계연도 말 + PIT_ANNUAL_LAG_DAYS (기본 90일)
    - 분기/반기(11012/11013/11014): 분기 말 + PIT_QUARTER_LAG_DAYS (기본 45일)
    반환: 'YYYY-MM-DD'
    """

def get_metrics_asof(conn, corp_code: str, as_of_date: str) -> Dict[str, float]:
    """as_of 재무 + as_of 시세로 PER/PBR/ROE 등 재계산.
    calculated_metrics를 그대로 읽지 않고 PIT 재무로 다시 계산해 누수를 막는다."""
```

**데이터 출처**: `disclosed_at`은 우선 `dart_disclosures.rcept_dt`(공시 접수일)를
`(corp_code, bsns_year, reprt_code)`로 매칭해 채운다. 매칭 실패 시
`estimate_disclosed_at()` lag fallback. (보고서 종류 코드는 02-외부-API-가이드.md §1 참조)

**불변식(invariant)**: 어떤 `as_of_date T`에 대해서도, 반환된 모든 재무 행의
`disclosed_at <= T` 가 항상 성립한다. (이게 깨지면 미래참조 누수 → 테스트로 강제)

### 4.2 `execution_model.py` — 체결·비용 (순수함수)

**책임**: 한국 시장의 현실적 거래비용과 체결 가능성을 모델링한다. DB 접근 없음.

**핵심 함수**
```python
from typing import Optional

def sell_tax_rate(trade_date: str) -> float:
    """매도 시 증권거래세율(농특세 포함). 시점별 상이.
    - 2022년까지: 0.23%
    - 2023년:     0.20%
    - 2024년:     0.18%
    - 2025년 이후:0.15%
    (코스닥/코스피 동일 가정. 실제 시행령 값으로 config에서 조정)"""

def apply_costs(side: str, price: float, qty: int, trade_date: str,
                commission_rate: float, slippage_bps: float) -> Dict[str, float]:
    """매수/매도 체결 시 실수령/실지불 금액 계산.
    - 공통: 수수료 commission_rate (양방향)
    - 매도: + 증권거래세 sell_tax_rate(trade_date)
    - 공통: 슬리피지 slippage_bps (불리한 방향)
    반환: {'fill_price', 'gross', 'commission', 'tax', 'slippage', 'net'}"""

def round_to_tick(price: float) -> int:
    """한국 주식 호가단위로 반올림.
    예: 2,000원 미만 1원 / 5,000 미만 5원 / 20,000 미만 10원 /
        50,000 미만 50원 / 200,000 미만 100원 / 500,000 미만 500원 / 그 이상 1,000원
    (2023.1 개편 기준; 정확한 구간표는 config.TICK_TABLE)"""

def is_tradable(side: str, ohlcv: Dict, prev_close: int,
                min_value_krw: int = 100_000_000) -> bool:
    """해당 봉에서 체결 가능한지.
    - 거래정지/거래없음: volume == 0 → False
    - 상한가 매수 불가: side=='buy' and close >= 상한가(prev_close*1.30) and high==low → False
    - 하한가 매도 불가: side=='sell' and close <= 하한가(prev_close*0.70) and high==low → False
    - 유동성 부족: 당일 거래대금 < min_value_krw → False"""
```

**설계 메모**: 상·하한가는 ±30%(2015년 이후), 호가단위는 2023.1 개편 기준을
`config.py`에 표로 둔다. 세율은 시점 함수로 둬서 과거 구간 백테스트가 그 시점 세율을 쓰게 한다.

### 4.3 `validation_harness.py` — 과최적화 방지

**책임**: IS/OOS 분리와 다중검정 보정으로 "진짜 알파"와 "노이즈 적합"을 구분한다.

**핵심 함수**
```python
from typing import List, Dict, Tuple

def split_is_oos(window_start: str, window_end: str,
                 oos_fraction: float = 0.3) -> Tuple[Tuple[str, str], Tuple[str, str]]:
    """기간을 앞 IS / 뒤 OOS 로 분할. 반환: ((is_start,is_end),(oos_start,oos_end))"""

def deflated_sharpe_ratio(observed_sharpe: float, n_trials: int,
                          n_obs: int, skew: float = 0.0, kurt: float = 3.0) -> float:
    """Bailey & López de Prado Deflated Sharpe Ratio.
    n_trials개 전략을 시도했을 때 관측 Sharpe가 우연이 아닐 확률(0~1).
    다중검정으로 부풀려진 최대 Sharpe 기대값을 차감해 보정한다."""

def passes_gate(is_metrics: Dict, oos_metrics: Dict,
                deflated_sr: float, min_dsr: float = 0.95) -> Tuple[bool, str]:
    """옵티마이저 반영 게이트.
    통과 조건(전부 충족):
      - oos_metrics['cagr'] > 0 (OOS에서도 수익)
      - oos_metrics['sharpe'] >= is_metrics['sharpe'] * 0.5 (IS 대비 과도한 붕괴 없음)
      - deflated_sr >= min_dsr
    반환: (통과여부, 사유 문자열)"""
```

**연동**: `n_trials`는 `simulation_runner`가 탐색한 파라미터 공간 크기를 넘긴다.
`passes_gate`가 False면 `algo_optimizer`는 해당 파라미터를 반영하지 않고 사유를 로깅한다.

**책임**: 현재 데이터로는 생존편향을 완전 제거할 수 없으므로, 그 크기를 추정·경고한다.

```python
    반환: {'delisted_ratio', 'estimated_cagr_haircut_pct', 'warning'}"""

def apply_haircut(reported_cagr: float, haircut_pct: float) -> float:
    """보고 CAGR에 보수적 haircut 적용."""
```

**원칙**: 이 모듈은 수익률을 *낮추는* 방향으로만 작동한다. 리포트에 한계를 명시한다

## 7. 데이터 흐름 (point-in-time 보장)

```
as_of T 시점 1회 백테스트:
  universe(T)   = price_data에 T 거래기록이 있는 종목
                  (생존편향: 소멸 종목은 자연 제외됨 → bias_report로 경고)
  financials(T) = get_financials_asof(conn, corp, T)   ← disclosed_at <= T 게이트
  metrics(T)    = get_metrics_asof(...)                ← PIT 재무로 재계산
  screening(T)  = 기존 전략 함수에 metrics(T) 주입
  진입/청산     = is_tradable() 통과 시 apply_costs(round_to_tick(price)) 로 체결
  성과          = split_is_oos로 OOS 구간만 CAGR/MDD/Sharpe 산출
  보정          = deflated_sharpe_ratio(n_trials) → passes_gate
  저장          = simulation_runs(oos_cagr, deflated_sharpe, gate_passed, ...)
```

## 8. 엣지 케이스 / 오류 처리

| 상황 | 처리 |
|------|------|
| `disclosed_at` 미상 | `estimate_disclosed_at()` lag fallback + 로깅 |
| 상한가에서 매수 시그널 | 체결 스킵(미체결) → 다음 거래일 재시도 또는 패스 |
| OOS 구간 데이터 부족 | run을 `gate_passed=0, gate_reason='insufficient_oos'`로 마킹, 옵티마이저 제외 |
| 세율 시점 변화 | `sell_tax_rate(trade_date)`로 그 시점 세율 적용 |
| 유동성 부족 소형주 | `is_tradable` False → 유니버스 제외 |
| PIT 재계산 결과 결손 | 해당 종목 스킵(점수 미산출), 전체 파이프라인은 계속 |

## 9. 테스트 전략 (TDD)

| 대상 | 테스트 핵심 |
|------|-------------|
| `execution_model` | 골든값 테이블 단위테스트(수수료/세금/슬리피지/호가/체결가능성) |
| `point_in_time` | **누수 차단 단언**: "2024 사업보고서가 2024-02-01 백테스트에 보이면 FAIL" |
| `validation_harness` | 과최적 시그니처 합성전략이 `passes_gate=False`로 걸러지는지 |
| `bias_report` | 소멸 종목이 있는 합성 유니버스에서 haircut > 0 |
| **회귀 특성테스트** | 신뢰성 적용 후 동일 전략의 보고 CAGR이 *하락*해야 정상 (부풀린 수익 현실화) |

---

## 10. 성공 기준 (Definition of Done)

1. 모든 백테스트가 `point_in_time` 게이트를 경유하며, 미래참조 누수 테스트가 통과한다.
2. 체결이 `execution_model`을 경유하며, 비용 적용 후 CAGR이 적용 전보다 낮다(특성테스트 통과).
3. `simulation_runs`에 `oos_cagr`/`deflated_sharpe`/`gate_passed`가 기록된다.
4. `algo_optimizer`가 `passes_gate=False` 파라미터를 반영하지 않고 사유를 로깅한다.
5. 일일 리포트에 생존편향 경고와 haircut이 표기된다.
6. 신규 테스트 전부 통과, 기존 테스트 회귀 없음.
